# 基于 Agent 的自动求职系统项目介绍与简历说明

## 1. 项目概述

本项目是一个面向 **AI Engineer / 算法工程师校招与实习岗位** 的自动化求职 Agent 系统。项目核心目标是让 Agent 像一个具备自主行动能力的“岗位搜集助手”一样，围绕目标岗位进行策略规划、检索执行、网页抓取、岗位分析、去重整理与结果导出，最终生成结构化的岗位数据，帮助求职者更高效地完成职位搜集与筛选。

与普通的爬虫或关键词检索脚本不同，这个项目引入了 **Plan-Act-Observe-Reflect** 的 Agent 工作方式：

- `Plan`：根据当前覆盖情况和历史搜索效果，动态制定下一轮搜索方向。
- `Act`：执行搜索、直爬、网页抓取与岗位分析。
- `Observe`：统计本轮新增岗位、来源分布、地域分布、领域覆盖情况。
- `Reflect`：反思当前策略效果，决定继续执行还是重新规划。

当前项目的默认目标是自动收集 **50 条 AI 校招/实习岗位**，并导出为 `JSON` 与 `CSV` 两种格式。

## 2. 项目解决的问题

传统找 AI 岗位时，常见问题包括：

- 招聘信息分散在多个平台，人工搜索成本高。
- 不同平台页面结构不一致，信息抽取困难。
- 搜索结果中混有大量聚合页、资讯页、无效页，人工甄别耗时。
- 同一岗位可能在多个渠道重复出现，需要二次清洗。
- 只靠固定关键词检索，容易造成来源、城市、方向覆盖不足。

本项目通过 Agent 化的搜索与反思机制，尽量解决上述问题：

- 自动规划多轮搜索策略，而不是只跑一轮固定关键词。
- 同时结合搜索引擎检索与目标站点直爬，提高召回率。
- 使用 LLM 判断岗位相关性并抽取结构化字段。
- 对 URL 与“公司 + 职位名”进行双重去重。
- 对岗位来源、地域、方向做覆盖率统计，驱动下一轮策略调整。

## 3. 核心架构设计

项目采用 **LangGraph StateGraph** 构建状态机式 Agent 流程，整体拓扑为：

```text
strategist -> executor -> reflector
                    ^         |
                    |         v
                    +---- replan / continue
```

### 3.1 三个核心节点

#### Strategist

战略规划节点，职责包括：

- 读取当前已收集岗位数量、覆盖率信息、历史反思记录、已使用关键词。
- 基于覆盖缺口生成 3-5 个新的“战略子目标”。
- 避免重复搜索已经尝试过的方向。
- 引导后续执行节点在来源、地域、AI 子方向上进行更均衡的探索。

#### Executor

执行节点，是整个系统最核心的部分。它并不是完全依赖 LLM 自主工具调用，而是采用 **“代码驱动执行 + LLM 辅助生成查询”** 的混合模式：

- 先基于模板规则生成一批高可靠搜索 query。
- 再让 LLM 补充 2-3 个更有针对性的关键词。
- 调用搜索工具获取候选链接。
- 调用直爬模块绕过搜索引擎，直接从已知招聘站点挖掘详情页。
- 对候选 URL 进行打分、去重、筛选和主机域名多样性控制。
- 并发抓取岗位页面并调用 LLM 分析是否属于目标岗位。
- 将有效岗位写入状态，并更新覆盖率统计。

这种设计兼顾了两点：

- 保留 Agent 的“动态规划与自适应搜索能力”。
- 避免让 LLM 直接控制所有执行细节，从而提升稳定性、可控性和效率。

#### Reflector

反思节点负责在每轮执行后判断当前策略是否有效，主要输出：

- 本轮收益高、中、低的判断。
- 哪类关键词或来源更有效。
- 哪些尝试失败较多。
- 当前最大的覆盖率缺口是什么。
- 是否需要重新规划下一轮策略。

当出现以下情况时，系统会优先重新规划：

- 本轮新增岗位为 0。
- 连续若干轮收益较低。
- 达到固定轮次间隔后主动 replan。
- 发现来源或方向覆盖失衡。

## 4. 项目工作流

下面是项目的一次完整运行流程：

### 步骤 1：初始化运行环境

程序入口在 `main.py`，启动后会：

- 加载 `.env` 中的 DeepSeek API 配置。
- 构建 LangGraph 状态图。
- 初始化 AgentState。
- 启动整轮求职搜索流程。

### 步骤 2：Strategist 生成搜索计划

Agent 根据当前进度生成多条搜索子目标，例如：

- 补充某些城市的岗位覆盖。
- 补充大模型、推荐、CV、NLP 等细分方向。
- 更多关注牛客网、实习僧、官方招聘页、高校就业网等稳定来源。

注意这里生成的是“战略方向”，不是最终 query。

### 步骤 3：Executor 生成 query 并发起搜索

执行节点先用规则模板生成一批 query，例如：

- 公司名 + 岗位方向 + 校招/实习
- 城市 + AI算法工程师 + 校园招聘
- 牛客网 / 实习僧 + 方向词

再让 LLM 根据本轮目标和历史失败信息补充查询词。

然后依次调用 `web_search` 进行搜索，当前实现支持：

- DuckDuckGo HTML
- Bing
- Baidu
- Sogou
- SerpAPI（可选，需配置 Key）

### 步骤 4：直接爬取已知招聘站点

为了降低搜索引擎限流、验证码和空结果带来的影响，项目还会直接爬取一些更稳定的来源：

- 实习僧
- 牛客网
- 国家大学生就业服务平台相关域名
- 部分高校就业站点

这一步可以理解为“搜索引擎召回 + 垂直站点直爬”的双通路召回策略。

### 步骤 5：候选 URL 筛选与排序

项目不会把所有搜索结果都送去抓取，而是先进行一轮工程化筛选：

- 跳过已经处理过的 URL。
- 过滤明显无效域名或低价值页面。
- 对候选链接做相关性打分。
- 给职位详情页模式更高分。
- 限制单一域名的链接数量，避免来源过于集中。

最终每轮只保留一部分高价值 URL 进入抓取分析阶段。

### 步骤 6：抓取页面内容

`fetch_page` 工具会根据站点特征自动选择抓取方式：

- 对静态 HTML 页面，使用 `requests + BeautifulSoup` 快速抓取。
- 对 BOSS、猎聘、拉勾、智联、腾讯招聘等强依赖前端渲染的网站，使用 `Playwright` 渲染后抓取。

同时加入了若干工程优化：

- 浏览器实例复用，减少重复启动成本。
- 域名失败计数与运行时黑名单，避免无意义重试。
- 验证码页识别，减少将无效内容送给 LLM。

### 步骤 7：LLM 分析岗位相关性并抽取结构化信息

抓取到正文后，`analyze_job` 会调用 DeepSeek 模型完成两件事：

- 判断该页面是否为 AI 相关岗位详情页。
- 提取结构化字段，包括：
  - 职位名称
  - 公司名称
  - 工作地点
  - 薪资
  - 技术标签
  - 技能要求摘要

如果页面只是聚合页、导航页、资讯页，系统会直接丢弃。

### 步骤 8：去重、统计、反思

新增岗位进入状态前，会经过两层去重：

- `job_url` 去重
- `company + title` 组合去重

随后系统会更新：

- 来源分布
- 城市分布
- AI 方向分布
- 公司分布
- 本轮新增数量

Reflector 根据这些信息决定：

- 继续执行当前计划
- 还是回到 Strategist 重新规划

### 步骤 9：导出结果

任务结束后，系统会在 `output/` 目录输出：

- `jobs.json`
- `jobs.csv`

当前仓库中的样例输出已经达到 **50 条岗位数据**。

## 5. 用到的技术栈

## 5.1 Agent 与流程编排

- **LangGraph**
  - 用于构建状态机式 Agent 流程。
  - 实现 `strategist -> executor -> reflector` 的多轮循环。
- **LangChain Core / Tools**
  - 用于封装 `web_search`、`fetch_page`、`analyze_job`、`get_coverage_report` 等工具接口。

## 5.2 大模型与接口层

- **DeepSeek Chat**
  - 用于策略规划、关键词生成、岗位分析、效果反思。
- **OpenAI Compatible API**
  - 通过 `langchain_openai.ChatOpenAI` 和 `openai.OpenAI` 接入 DeepSeek。

## 5.3 搜索与抓取

- **requests**
  - 发起搜索引擎请求、直爬招聘站点页面、抓取静态网页内容。
- **BeautifulSoup4**
  - 解析 HTML，抽取搜索结果与岗位正文。
- **Playwright**
  - 抓取需要 JavaScript 渲染的页面。

## 5.4 数据建模与导出

- **Pydantic v2**
  - 定义岗位数据结构 `Job`，保证输出字段统一。
- **pandas**
  - 将岗位结果导出为 CSV。
- **json**
  - 结构化存储岗位结果和覆盖率信息。

## 5.5 工程辅助

- **python-dotenv**
  - 管理环境变量。
- **ThreadPoolExecutor**
  - 并发抓取多个 URL，提升处理效率。
- **Counter / TypedDict / Pathlib**
  - 统计、状态建模与路径管理。

## 8. 文件结构与功能说明

### 8.1 根目录文件

#### `main.py`

项目主入口，负责：

- 加载环境变量。
- 构建并运行 Agent 图。
- 将最终结果解析为 `Job` 对象。
- 执行结果去重。
- 导出 `JSON` 与 `CSV`。
- 输出汇总统计信息。

#### `README.md`

项目说明文档，包含项目目标、架构简介、运行方式和输出格式。

#### `requirements.txt`

项目依赖列表，包括：

- langgraph
- langchain-openai
- langchain-core
- openai
- requests
- beautifulsoup4
- pydantic
- python-dotenv
- pandas
- playwright

#### `.env`

环境变量配置文件，主要用于配置：

- `DEEPSEEK_API_KEY`
- `DEEPSEEK_BASE_URL`
- 可选的 `SERPAPI_API_KEY`

#### `Inspire 校招 AI Engineer Homework.pdf`

项目题目或作业说明材料，可理解为项目需求背景文件。

### 8.2 `agent/` 目录

#### `agent/graph.py`

整个项目最核心的调度文件，负责：

- 定义 LangGraph 工作流。
- 创建 LLM 实例。
- 实现 strategist、executor、reflector 三个节点。
- 处理 query 生成、URL 评分、并发处理、覆盖率计算、流程跳转等核心逻辑。

#### `agent/state.py`

定义 `AgentState` 状态结构，包括：

- 消息历史
- 已收集岗位
- 搜索过的 query
- 已处理 URL
- 当前计划
- 反思记录
- 覆盖率
- 轮次与状态

#### `agent/prompts.py`

存放 Strategist 和 Reflector 的提示词模板，用于：

- 规划下一轮搜索策略
- 分析本轮执行效果并给出改进建议

### 8.3 `tools/` 目录

#### `tools/web_search.py`

搜索工具模块，负责：

- 调用多个搜索引擎进行岗位检索。
- 合并搜索结果。
- 去除重复 URL。
- 识别结果来源网站。
- 处理限流与验证码场景。

#### `tools/page_fetcher.py`

网页抓取模块，负责：

- 判断页面是否需要 JS 渲染。
- 静态抓取页面正文。
- 使用 Playwright 渲染动态站点。
- 检测验证码页和异常页。
- 对失败域名进行运行时黑名单管理。

#### `tools/job_analyzer.py`

岗位分析模块，负责：

- 使用 LLM 判断岗位是否相关。
- 识别是否为详情页而不是聚合页。
- 抽取岗位结构化字段。
- 统一补充 `job_url` 和 `source`。

#### `tools/direct_crawl.py`

站点直爬模块，负责：

- 直接从已知招聘平台抓取岗位详情 URL。
- 弥补搜索引擎召回不稳定的问题。
- 当前重点支持实习僧、牛客网、NCSS 和部分高校就业站点。

#### `tools/coverage_report.py`

覆盖率统计模块，负责：

- 保存当前覆盖率分析结果。
- 保存已收集岗位列表，供异常恢复使用。
- 提供覆盖率查询工具给 Agent 使用。

### 8.4 `models/` 目录

#### `models/job.py`

定义岗位数据模型 `Job`，主要字段包括：

- `title`
- `company`
- `location`
- `salary`
- `tech_tags`
- `requirements`
- `source`
- `job_url`

同时提供 `dedup_key()` 方法，方便按“公司 + 职位名”去重。

### 8.5 `utils/` 目录

#### `utils/dedup.py`

去重工具模块，负责：

- 基于岗位链接去重。
- 基于公司名和职位名组合去重。

### 8.6 `output/` 目录

#### `output/jobs.json`

结构化岗位输出文件，适合程序进一步处理或二次分析。

#### `output/jobs.csv`

表格化岗位输出文件，适合人工查看、筛选或导入 Excel。

## 9. 项目亮点总结

这个项目的亮点并不只是“把招聘网页爬下来”，而是将 **Agent 思维、工程化执行、LLM 语义理解、结构化输出** 串成了一个完整闭环：

1. 采用 LangGraph 构建可迭代、可反思的 Agent 工作流，而不是一次性脚本。
2. 使用“规则模板 + LLM 补充”的混合 query 生成策略，兼顾可控性与灵活性。
3. 同时结合搜索引擎检索和垂直站点直爬，提高岗位召回率。
4. 面向真实网页环境，处理了 JS 渲染、验证码页、失败域名黑名单等工程问题。
5. 利用 LLM 完成岗位相关性判断和结构化字段抽取，降低人工筛选成本。
6. 内置覆盖率分析和反思机制，支持动态 replan，而不是固定关键词暴力搜索。

### 基于 Agent 的自动求职系统  `2026.03 - 2026.04`  个人项目

- **项目简介**：面向 AI Engineer / 算法工程师校招与实习场景，设计并实现基于 LangGraph 的自主求职 Agent，系统可自动完成搜索规划、多源岗位召回、网页抓取、岗位理解、去重清洗与结构化导出，形成从“岗位发现”到“结果整理”的完整闭环。
- **技术栈**：LangGraph、LangChain、DeepSeek、OpenAI Compatible API、Python、Requests、BeautifulSoup、Playwright、Pydantic、Pandas
- **项目亮点**：
  1. **设计 PAOR Agent 工作流**：基于 `Plan-Act-Observe-Reflect` 思路实现三阶段状态机，构建 `strategist -> executor -> reflector` 的多轮迭代流程，使系统能够根据搜索收益和覆盖率自动调整下一轮求职策略。
  2. **实现“规则模板 + LLM 补充”的搜索规划机制**：在执行阶段先基于城市、公司、方向词生成高可靠 query，再利用 LLM 结合历史失败信息补充检索词，提升岗位检索的针对性与多样性。
  3. **构建多源岗位召回链路**：同时接入 DuckDuckGo、Bing、Baidu、Sogou 等搜索引擎，并增加对实习僧、牛客网、NCSS、高校就业站点的直爬逻辑，降低单一搜索源失效带来的召回损失。
  4. **完成网页抓取与动态页面适配**：针对静态页面使用 `requests + BeautifulSoup` 进行正文提取，针对 BOSS、猎聘、拉勾、智联等动态站点引入 Playwright 渲染抓取，并增加验证码检测、失败域名黑名单和浏览器复用等机制，提升抓取稳定性。
  5. **实现 LLM 驱动的岗位理解与结构化抽取**：利用 DeepSeek 对职位页面进行相关性判断，过滤聚合页、资讯页和无效页面，并抽取职位名称、公司、地点、薪资、技术标签、技能要求等字段，统一输出为结构化岗位对象。
